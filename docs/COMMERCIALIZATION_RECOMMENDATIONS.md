# NutriSort AI 상용화 권장 사항

> 혈당 관리 앱 상용 서비스 전 필수·권장 항목 정리

---

## 1. 법적·규제 (필수)

### 1.1 의료·건강 앱 관련
- **현재**: “혈당 관리”, “섭취 순서” 등 건강 정보 제공 → **의료기기 아님**이지만 **건강정보 제공자**에 해당할 수 있음.
- **권장**:
  - 앱 내 고지 문구 추가: *"본 서비스는 의료 행위가 아니며, 진단·치료를 대체하지 않습니다. 당뇨 등 질환 관리 시 반드시 의료진과 상담하세요."*
  - 식품/영양 정보는 “참고용”임을 명시 (예: 식단 기록·AI 분석 결과는 참고용).

### 1.2 개인정보처리방침·이용약관
- **필수**: 서비스 도메인에 **개인정보처리방침**·**이용약관** 페이지 (또는 링크) 제공.
- 수집 항목: 이메일, (선택) 프로필, 식단 기록·사진(저장 시), 기기 정보(로그용).
- 보관 기간, 제3자 제공(Google/Firebase 등), 이용자 권리(열람·삭제·동의 철회) 명시.
- **권장**: 회원가입/최초 이용 시 “이용약관·개인정보처리방침에 동의합니다” 체크 후 가입 가능하도록 연동.

### 1.3 연령 및 이용 제한
- 만 14세 미만 등 제한이 있다면 이용약관에 명시하고, 가입 시 생년월일 수집·검증 고려.

---

## 2. 보안·비용 관리 (필수)

### 2.1 API·비용 제어
- **Gemini API**: 사용량 상한·알림 설정(Google Cloud 콘솔). 일일/월간 예산 알림 권장.
- **Firebase**: Blaze 플랜 사용 시 예산·알림 설정. Firestore 읽기/쓰기 규칙 재검토.
- **권장**: 유료 구독자와 무료·게스트에 대해 **일일/월간 분석 횟수 제한**을 다르게 두고, 제한 초과 시 안내 메시지 + 업그레이드 유도.

### 2.2 Rate limiting·남용 방지
- IP·계정별 **분당/일일 요청 제한** (예: 분당 5회, 일 50회) 적용.
- Streamlit 단에서 `st.session_state`로 간단 카운터 구현 가능. 상용화 단계에서는 **백엔드(Cloud Functions/FastAPI 등)에 rate limit** 두는 것을 권장.

### 2.3 시크릿·환경 분리
- **이미 잘 되어 있음**: `secrets.toml` 미커밋.
- **권장**: 개발/스테이징/운영 환경 분리. Streamlit Cloud는 앱별로 다른 secrets 사용 가능하므로 “프로덕션” 앱과 “개발” 앱을 분리해 운영.

### 2.4 로깅·모니터링
- **에러 로깅**: `try/except` 구간에서 구조화된 로그(날짜, 사용자 구분자, 에러 메시지) 출력. Streamlit Cloud 로그만으로도 초기에는 가능.
- **권장**: 장기적으로 Sentry, Logtail 등으로 에러 수집·알림 설정.

## 3. 수익화 준비

### 3.1 요금제 설계 예시
- **무료**: 일 2~3회 분석, 기본 기록 보관 7일 등.
- **유료(월/연 구독)**: 무제한 분석, 장기 기록 보관, 목표별 상세 리포트 등.
- 게스트 “광고 보고 1회 충전”은 현재 구조 유지 가능. 실제 상용화 시 **Google AdMob 리워드 광고**로 교체 권장.

### 3.2 결제 연동
- **Stripe** 또는 **토스페이먼츠** 등으로 구독(월/연) 결제 연동.
- Firebase Auth UID와 결제 구독 상태를 Firestore(예: `users/{uid}/subscription`)에 저장하고, 앱 진입 시 “유료 권한” 여부로 분석 횟수·기능 제한.

### 3.3 이용약관에 수익화 반영
- 무료/유료 차이, 환불 정책, 구독 해지 방법을 이용약관에 명시.

## 4. 안정성·사용자 경험

### 4.1 에러 메시지·재시도
- **이미 적용**: Gemini 503 재시도, 게스트 횟수 복구.
- **권장**: “일시적 오류입니다. 잠시 후 다시 시도해 주세요.” 등 사용자 친화적 문구 통일. 필요 시 “다시 시도” 버튼 제공.

### 4.2 데이터 백업
- Firestore **자동 백업**(Google Cloud 스케줄 내보내기) 설정 권장. 사용자 식단 기록은 복구 가능해야 함.

### 4.3 다운타임·점검 공지
- 유지보수 시 Streamlit 앱 대신 “점검 중” 페이지로 대체하거나, 사이드바/상단에 “오늘 00:00~02:00 점검 예정” 등 안내.

---

## 5. 코드·운영 품질

### 5.1 의존성 버전 고정
- `requirements.txt`에 버전 명시로 배포 재현성 확보.
  - 예: `streamlit>=1.28.0`, `google-genai>=0.2.0`, `firebase-admin>=6.0.0`, `Pillow>=10.0.0`, `requests>=2.28.0`, `pandas>=2.0.0`

### 5.2 설정·URL 중앙화
- 앱 URL, 리다이렉트 URI 등을 **환경 변수 또는 config**로 빼기. 예: `BASE_URL`, `REDIRECT_URI` → 코드 상 하드코딩 제거.

### 5.3 앱 분리(선택)
- `app.py`가 크므로, 인증·분석·UI·설정을 모듈로 나누면 유지보수·테스트에 유리.

---

## 6. 브랜딩·고객 지원

### 6.1 앱 내 고객센터
- “문의하기” 메뉴: 이메일 주소 또는 문의 폼 링크. 이용약관/개인정보처리방침 링크 항상 노출.

### 6.2 버전·업데이트 안내
- 앱 하단 또는 설정에 “버전 1.0.0” 표기. 중요 업데이트 시 한 번만 뜨는 “업데이트 안내” 모달 권장.

### 6.3 스토어 출시(선택)
- PWA만으로도 서비스 가능. 추후 **Google Play / Apple App Store**에 웹뷰 앱으로 패키징하면 검색·설치 경험 개선 가능 (각 스토어 정책·개인정보처리방침 필수).

---

## 7. 플랫폼·스케일

### 7.1 Streamlit Cloud
- 무료 티어: 앱 슬립, 리소스 제한 있음. 상용 트래픽이 늘면 **Streamlit Cloud 유료** 또는 **자체 호스팅**(AWS/GCP 등에 Streamlit 배포) 검토.

### 7.2 장기 이전 옵션
- 프론트는 React/Next 등으로, 백엔드는 FastAPI 등으로 분리하면 확장·캐싱·결제 연동이 유연해짐. 당장 필수는 아니고, 트래픽·기능이 커질 때 단계적으로 검토.
